Имена свойств

Value

Изменения касаются некоторых классов провайдеров управляющих элементов EFPxxx и диалогов в пространстве имен FreeLibSet.Forms и классов удаленного пользовательского интерфейса в пространстве имен FreeLibSet.RI

Простые имена свойств - вводимых значений относятся к не-nullable значениям типа T. Например, Text, Value, IntValue и т.д. Соответствующие управляемые свойства DepValue<T> имеют имена с суффиксом "Ex", например, TextEx, ValueEx, IntValueEx (это правило не изменилось по сравнению с версией 1.1). Если типом значения является структура, а не класс, и управляющий элемент поддерживает ввод пустых значений, то имеются дополнительные свойства с префиксом "N" и имеющих тип Nullable<T>, например, NValue, NIntValue. Соответствующие управляемые свойства имеют имена NValueEx, NIntValueEx. Если пустое struct-значение может быть представлено без использования nullable-типа, например, YearMonth.Empty, то используются простые имена свойств без префикса "N", например, YM, YMEx.

Для задания диапазона допустимых значений используются свойства Minimum, Maximum. Управляемых свойств MinimumEx, MaximumEx нет. Эти свойства могут иметь Nullable-тип, где null задает отсутствие ограничения. Префикс "N" не используется. Обычно, ограничения обрабатываются на уровне EFPControl. Для некоторых элементов, например, в IntEditBox, свойства обрабатываются также на уровне управляющего элемента (используются для прокрутки стрелочками).

Для "серых" значений, отображаемых, когда свойство Enabled=false, или ReadOnly=true, используются существующие правила именования свойств. Разрешающее свойство типа boolean имеет имя с префиксом "AllowDisabled", например, AllowDisabledValue. Для таких свойств нет управляемых свойств с суффиксом "Ex". Само "серое" значение хранится в свойстве с префиксом "Disabled", например, DisabledValue. Для него имеется парное управляемое свойство DisabledEx. Если есть парные nullable-свойства, то запрещенные значения также дублируются, то есть должны быть свойства DisabledNValue и DisabledNValueEx. Однако разрешающее свойство не дублируется, то есть свойства AllowDisabledNValue нет. Удаленный интерфейс по прежнему не поддерживает серые значения.

Если элемент может принимать пустое значение (TextBox, DateTimeBox, DateRangeBox, IntEditBox, ...), то используются следующие свойства

- UIValidateState CanBeEmptyMode {get;set;}. По умолчанию - Error. Можно установить Warning или Ok.

- bool CanBeEmpty {get;set;}. Если false (по умолчанию), то будет выдаваться сообщение об ошибке. Дублирует CanBeEmptyState

- Управляемое свойство DepValue<bool> IsNotEmptyEx {get;}. Только выход, возвращающий true, если в поле есть непустое значение. Свойство можно использовать в качестве предусловия для валидатора UIValidator.

В ExtForms использовалось свойство WarningIfEmpty. Оно заменяется на CanBeEmptyMode.

Управляемые свойства CanBeEmptyEx и WarningIfEmptyEx упраздняются. Они могут быть легко реализованы с помощью UIValidator, если в качестве условия использовать свойство IsNotEmptyEx.

Другие свойства для проверки

Остаются только свойства Minimum и Maximum. В основном, потому что они нужны еще и для прокрутки числовых значений. Управляемые свойства (MinumumEx и MaximumEx) убираются. Свойство WarningIfOutOfRange убирается, можно задавать только ошибку, а не предупреждение. Чтобы задать предупреждение для попадания в диапазон, можно использовать валидатор с методом DepTools.InRangeEx() в качестве входа.

Свойства Error/WarningRegExpattern/Message убираются. Проверку можно реализовать валидатором с методом DepTools.RegexIsMatchEx().

Проверка в блоках диалога

Реализуется проверка с помощью UIValidator. Класс BaseInputDialog (или StandardDialog ?) содержит свойство Validators. Конкретные реализации диалогов содержат управляемые свойства. Например, DateTimeInputDialog содержит свойства NValueEx, ValueEx, NTimeEx и TimeEx. В отличие от управляющих элементов, эти свойства предназначены только для чтения. Также есть свойство IsNotEmptyEx.

В ExtForms вносятся такие же изменения. Например, класс DateTimeInputDialog содержит список Validators и управляемые свойства и IsNotEmptyEx только для чтения.

Существующие события Validating убираются, т.к. их можно заменить на объекты UIValidator. Отличие: Обработчик UIValidator вызывается и при пустом текущем значении, если проверка не остановилась на CanBeEmpty=false. Можно подсоединить IsNotEmptyEx к предусловию.
